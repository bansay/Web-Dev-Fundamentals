Containers

Containers are one of the most important concepts to grasp in frontend web development. The web, and it's design + framework is essentially made up of rows and columns of information. The web has truly offered some bright alternative ways of looking at this not-so-riveting fact in recent years, with some super slick examples of web dev (thinking of sources like codepen.io, jsfiddle.net, jsbin.com, etc.), such examples that test the rules of the DOM. But, to test the rules effectively you must know the rules, right? Thus, knowing how information inherently displays in the DOM will help you write better HTML, CSS, + Javascript, and build better user experiences from it.

However, back in the day, there were unavoidable limitations to displaying data on a web page, that is before the div tag's were established - so basically the entire web was made up of tables. And we all know what tabular data looks like.

Since then, the web has vastly improved with powerful CSS properties and refined HTML elements to allow for greater control over how website data is visualized/displayed, and how it's marked up in the document outline.

Take any website and you'll find rows and columns. Like... Vogue, for an example:

What's the Point? Why does it matter?

As soon as you can start to see the web for what it is - a bunch of rows, and then columns inside those rows, to define more modules of data, which hold more rows and columns inside those, you'll quickly be able to build "modules" of data that can be reusable across multiple pages. It also particularly matters when you want to "collapse" the data in a nice manner, so that certain data can be displayed differently for smaller and larger devices.

A common practice before responsive web design was to give the entire page a content container, say div id="container" and then give that container a fixed width, like width: 1024px; margin: 0 auto, so that the page is always 1024 pixels and the site is optimized for most computer screens (at the time, keeping in mind). An example of this practice is as follows:

Example 1:

<style type="text/css"> #container { width: 1024px; margin: 0 auto; } </style> <body> <div id="container"> <div id="header"> <h1>Name of Site</h1> <div id="nav"> <ul> <li><a href="#">Link 1</a></li> <li><a href="#">Link 2</a></li> </ul> </div> </div> <div id="section"> <!-- main section content--> </div> <div id="footer"> <!-- main footer content--> </div> </div> </body> View Live Example of Code above (Example 1: Non Responsive Container)

This was generally considered a fine practice, as people were viewing web pages on a standard computer, with a standard width they could pretty much hit all the horses with at 1024. In fact, setting hard widths on pretty much everything was considered just fine.

But now, however, we have a lot of gadgets that connect to the web, including: tablets, iPhones, androids, widths of all kinds, ranging anywhere from 200px all the way to 2200 pixels and more. Lots of widths. Think of it literally to comparing various canvas widths (like the one you paint on, all the options)..

So, therefore, locking yourself into a fixed pixel ratio width of any element size is really not practiced much anymore, IMHO.

Additional (but important) info about Example 1:

The disadvantages of using lots of ID's, and why you should use more classes:

IDs are costly. In fact, if an ID and a class went back-to-back in a ball game, the ID scores 40 points for one shot, and the class scores 10 points for the same shot. What this basically means is that IDs will outweigh the strength of a class 4:1 (or close to), if they were to ever go against eachother, in trying to target the same selector. IDs can be however very powerful and useful to use, to isolate certain scopes of style and behavior, where necessary.

Why you should use classes far more often than you may think Is that classes are more negotiable over time. In case anything needs to change down the road, it's harder to back track from IDs, and more often than not you'd never want to change an ID after you've established it. So when you use an ID, ask yourself,

"am I targeting this element for any type of javascript and/or interactive advanced css3 goodness?"
If yes, sure make it an ID, but if not... Make it a class. Just Sayin' - You'll be glad you did, later.

The other reason is that elements of HTML can only have 1 ID. Just 1. On the other hand, elements can have lots of classes. Not tons - but you get the picture? When you assign an element an ID, it better be for super good reasons, like - this thing is so unique you are gonna target it with event-bindings in javascript. That's my rule of thumb.

Never use a div id="container"

More often than not, you'll need to work with multiple backgrounds for a web page that need to span 100% of the width of any row of content, for any browser width. And by only using one container, with a fixed width set in pixels, you limit yourself to one background for the parent of that "contained" area on a page. And if you're not following the DRY (Don't Repeat Yourself) method by repeating yourself in styles, then it's likely that whatever it is you're styling may be better suited as a class.

What to look forward to:

The practice explained in Example 1 has been replaced by the notion of treating each row of content as its own container, instead of the page as a container, and thus allowing the width of the rows to be relative to the width of the device looking at it. What does this mean? It's often regarded as "fluid", and I consider it a mixture of using percentages and understanding inherent widths of elements. You and you can even set a max-width and min-width and be compatibile with all modern browers + IE >= IE8. This example below also uses HTML5 markup, which uses up-to-date DOM elements that have essentially been founded over time, over common patterns in building html pages, etc.

Example 2:

<style type="text/css"> .name-of-page { background: url('img/small_steps/small_steps.png') repeat; } .container { max-width: 1100px; min-width: 300px; width: 95%; margin: 0 auto; } </style> <body class="name-of-page"> <header class="container"> <h1>Name of Site</h1> <nav> <ul> <li><a href="#">Link 1</a></li> <li><a href="#">Link 2</a></li> </ul> </nav> </header> <section class="container"> <article> <!-- main section content--> </article> </section> <footer class="container"> <!-- main footer content--> </footer> </body> View Live Example of Code above (Example 2: Responsive Container)

An important thing to mention is you MUST have the viewport turned on in the head:

<meta name="viewport" content="width=device-width, initial-scale=1">

Thankfully, boilerplates like HTML5 provide that for you.

A little off topic, but note the class on the body: If you give the body a class - this is a good place to jump from on any given page, if you're working with any type of unique styles or themes on a page. Also to keep in mind, if you're using a Content Managing System like WordPress - any CMS of this nature, it will automatically give the body a class - so even for projects not using a database, it is a good way to establish unique styles on any given page.

Benefits of treating each row of content as a container instead of only having one container

One of the biggest benefits is this process allows for different backgrounds for different rows of content, while also giving the content inside a container. It would require adding another container, a div is a great use case of this as a div has no extraordinary meaning on the document outline of a web page. We can wrap our content in a div and give it a container, while allowing the parent element to hold a background.

An example of this is as follows:

Example 3:

<body class="name-of-page"> <header> <div class="container"> <h1>Name of Site</h1> <nav> <ul> <li><a href="#">Link 1</a></li> <li><a href="#">Link 2</a></li> </ul> </nav> </div> </header> <section> <div class="container"> <article> <!-- main section content--> </article> </div> </section> <footer> <div class="container"> <!-- main footer content--> </div> </footer> </body>

And the corresponding CSS for Example 3 could look something like this:

<style type="text/css"> .container { max-width: 1100px; min-width: 300px; width: 95%; margin: 0 auto; } header { background: url('img/light_wool/light_wool.png') repeat; } section { background: url('img/congruent_pentagon/congruent_pentagon.png') repeat; } footer { background: url('img/small_steps/small_steps.png') repeat; } header .container, section .container, footer .container { background: white; border: solid 1px #e5e5e5; padding: 15px 30px; margin-bottom: 30px; } </style> View Live Example of Code above (Example 3: Responsive Containers with Backgrounds)

So, to sum it all up.

Responsive web design is awesome. The * {box-sizing} trick is also awesome (view page source on Example 3 to see what I mean). Multiple backgrounds that span 100% the width are pretty great. Using IDs is bad, usually, unless you can really really justify it. Web pages are made up of a bunch of rows and columns. And setting a fixed pixel size to a "main" container for a page is no longer adaptable enough for the variety of devices that can show webpages. Ttherefore, switching to a percentage width for a class called "container" is good. And using min- and max- widths can also be beneficial.
